第四章问答作业

  1. 请列举 SV39 页表页表项的组成，结合课堂内容，描述其中的标志位有何作用／潜在作用？

  1. R/W/X (bits 1–3)：控制页面的访问权限。当 R/W/X 全为 0 时，PTE 不指向物理页而是指向下一级页表,此设计使得多级页表结构紧凑，无需额外类型标记。

  2. U (bit 4) — 核心隔离机制：U = 1 表示用户态可访问；U = 0 时只能在内核态访问。这是实现用户/内核隔离的关键：内核页面的 PTE 的 U 位被清零，任何在用户态（U 模式）执行的内存访问指令，若访问到 U=0 的页面，都会触发异常，从而防止用户态代码直接访问内核内存。

  3. G (bit 5)：标记为 Global 的页表项在 TLB 中不会被进程切换时自动刷新。典型的应用场景：内核空间是所有进程共享的，若能将其标记为 Global，进程切换时只需刷新用户的 TLB 条目，可显著提升性能。

  4. A (bit 6) 与 D (bits 7)：硬件在使用页面时会自动置位，主要用于页置换算法的实现：

  • A (Accessed)：页面被访问过，OS 可用其追踪页面使用频率，实现 LRU 近似页置换。

  • D (Dirty)：页面被写入过，在换出页面到磁盘时，若 D = 0 说明页面与磁盘镜像一致，无需写回（节省 I/O）；若 D = 1 则需要写回磁盘。

  1. RSW (bits 8–9)：完全由软件自由定义，OS 可利用其存储扩展元信息，例如引用计数（用于 COW）、页是否被换出等。

  2. 53–10（共 44 位): PPN(物理页号),指向物理页帧的高 44 位地址

  1. 缺页相关问题
  1. 请问哪些异常可能是缺页导致的？

  • 页不存在：访问的线性地址对应的页表项（PTE）的 Present 位为 0。

  • 权限不足：访问操作（读/写/执行）与 PTE 中的权限位（如 R/W、U/S）冲突。

  • 保留位错误：PTE 中某些必须为 0 的位被设置为 1（较少见）。

  1. 发生缺页时，描述相关的重要寄存器的值（lab2中描述过的可以简单点）。

  • CR2：存放引发缺页的线性地址（虚拟地址），OS 通过它确定哪个页面缺失。

  • 错误码（Page Fault Error Code）：位于异常栈中，提供附加信息，包括：

    P = 0/1：异常是否由页不存在引起（0 表示缺页）。

    W/R = 0/1：引发异常的操作是读（0）还是写（1）。

    U/S = 0/1：异常发生时 CPU 处于内核态（0）还是用户态（1）。

  • EIP / RIP：指向引发异常的指令地址（OS 在完成缺页处理后需重新执行该指令）。

  • EFLAGS / RFLAGS：保存当前 CPU 状态

缺页有两个常见的原因，其一是 Lazy
策略，也就是直到内存页面被访问才实际进行页表操作。比如，一个程序被执行时，进程的代码段理论上需要从磁盘加载到内存。但是 os
并不会马上这样做，而是会保存 .text 段在磁盘的位置信息，在这些代码第一次被执行时才完成从磁盘的加载操作。

  1. 这样做有哪些好处？

  • 加快程序启动速度：无需等待全部代码从磁盘载入，按需加载可大幅缩短启动时间。

  • 节省物理内存：只加载实际执行到的页面，避免为大量可能永远不运行的代码（如错误处理路径、冷代码）浪费内存。

  • 减少磁盘 I/O：避免不必要的磁盘读取，尤其对于大型程序，通常仅执行一小部分代码。

  • 提高系统整体吞吐量：内存和 I/O 资源被更高效地利用，可同时运行更多进程。

此外 COW(Copy On Write) 也是常见的容易导致缺页的 Lazy 策略，这个之后再说。其实，我们的 mmap 也可以采取
Lazy 策略，比如：一个用户进程先后申请了 10G 的内存空间，然后用了其中 1M
就直接退出了。按照现在的做法，我们显然亏大了，进行了很多没有意义的页表操作。

  1. 请问处理 10G 连续的内存页面，需要操作的页表实际大致占用多少内存(给出数量级即可)？

采用典型 x86-64 四级页表（每页 4KB，每项 8 字节，每级索引 9 位）：

  • 虚拟地址空间连续 10 GB 对应页面数：
    10×230/4×210=2.5×220=2,621,440 个页面。

  • 每个页面需要一个 PTE（8 字节），PTE 总大小：约 20 MB。

  • 中间各级页表（页目录等）还需额外开销：对于连续区域，每 512 个 PTE 共享一个页表页（4KB），因此需要约 2,621,440/512≈5,120 个页表页，即 20 MB；加上顶层页表（PGD、PUD、PMD）约几十 KB。

  • 总内存占用数量级：几十 MB（实际约 20~25 MB）

  1. 请简单思考如何才能在现有框架基础上实现 Lazy 策略，缺页时又如何处理？描述合理即可，不需要考虑实现。

  • 分配虚拟地址时：只操作 VMA（虚拟内存区域）结构，记录区域大小、权限、文件映射信息（如偏移），不为该区域创建任何最终页表项，或者创建 PTE 时将 Present 位设为 0，并利用保留位存储 Lazy 标志。

  • 缺页处理：

    1. 缺页异常处理程序读取 CR2 获取故障地址。

    2. 查找该地址所属的 VMA → 检查是否为合法的 Lazy 区域。

    3. 若合法，则：

      • 分配一块物理页面（若为匿名页则清零；若为文件映射则从磁盘/文件系统读取相应内容）。

      • 在该进程的页表中建立映射（设置 PTE，Present=1，权限与 VMA 一致）。

      • 刷新 TLB 并返回，重新执行引发异常的指令。

    4. 若不合法，则向进程发送 SIGSEGV 或杀死进程。

缺页的另一个常见原因是 swap 策略，也就是内存页面可能被换到磁盘上了，导致对应页面失效。

  1. 此时页面失效如何表现在页表项(PTE)上？

当物理页面被换出到磁盘后，对应的 PTE 会做如下修改：

  • Present 位清零（表示该页面不在内存中）。

  • 其余位（原本存储物理页框号的位置）被重用来存放 交换区信息（swap entry），例如：

    • 磁盘块号（或 swap 分区内的偏移）。

    • 交换标识（区分不同 swap 设备/分区）。

    • 其他辅助标志（如页面被换出前是否为脏页等）。

此时，访问该虚拟地址将触发缺页异常，OS 根据 PTE 中存储的 swap entry 找到磁盘上的页面，换入内存并恢复 PTE 的正常映射。

  1. 双页表与单页表

为了防范侧信道攻击，我们的 os
使用了双页表。但是传统的设计一直是单页表的，也就是说，用户线程和对应的内核线程共用同一张页表，只不过内核对应的地址只允许在内核态访问。请结合课堂知识回答如下问题

  1. 单页表情况下，如何更换页表？

单页表下，每个进程拥有一张独立的页表（包含用户空间和内核空间的映射）。当进行进程切换时，操作系统需要将新进程的页表基址加载到 CR3 寄存器中（x86 架构），即执行 mov cr3, new_page_table_base。这会刷新 TLB（通常为全部条目，部分架构支持全局页保留），从而完成地址空间的切换。

  1. 单页表情况下，如何控制用户态无法访问内核页面？（tips:看看第一题最后一问）

利用页表项（PTE）中的 用户/超级用户位（U/S bit）。在 x86 架构中，PTE 的 bit 2 为 U/S 位：

  • 该位为 0 表示超级用户（内核态，CPL=0） 才能访问；

  • 该位为 1 表示用户态（CPL=3） 也可访问。

因此，内核页面的 PTE 将 U/S 位设为 0。当 CPU 处于用户态时访问这些页面会触发页错误（权限不足异常），而内核态访问则正常。这就实现了用户态无法访问内核页面的隔离。

  1. 单页表有何优势？（回答合理即可）

  • 上下文切换开销小：陷入内核（系统调用、中断）时无需切换页表，CR3 保持不变，避免了 TLB 完全刷新，提高了内核与用户态之间数据拷贝（如 copy_from_user）的效率。

  • 实现简单：不需要维护两套页表，也不需要为每个进程同步内核映射（双页表下需确保内核映射一致性或共享内核页表）。

  • 内存占用低：每个进程仅需一张页表，而非用户页表 + 内核页表（双页表即使共享内核部分，仍需独立的内核页表基址指针）。

  • 内核访问用户内存方便：由于用户虚拟地址在当前页表中直接有效，内核可直接解引用用户指针，无需进行临时的地址映射。

  1. 双页表实现下，何时需要更换页表？假设你写一个单页表操作系统，你会选择何时更换页表（回答合理即可）？

双页表（如 KPTI）场景下：通常采用用户页表与内核页表分离的方式。需要更换页表的时机包括：

  • 用户态 → 内核态：如系统调用、中断、异常发生时，切换到内核页表（CR3 指向内核页表）。

  • 内核态 → 用户态：返回用户空间前，切换回用户页表。

  • 进程切换时：切换运行进程，需要更换成新进程的用户页表（同时内核页表通常全局共享，但若采用每进程内核页表则也需更换）。

若设计一个单页表操作系统，我会选择在以下时机更换页表：

  • 仅进程切换时：因为单页表下内核与用户共享相同的页表映射，用户态与内核态之间的切换无需改变 CR3。因此，唯一需要更换页表的场景是调度器决定切换到另一个进程时，将 CR3 设置为新进程的页表基址。

stride 算法深入

stride 算法原理非常简单，但是有一个比较大的问题。例如两个 pass = 10 的进程，使用 8bit 无符号整形储存
stride， p1.stride = 255, p2.stride = 250，在 p2 执行一个时间片后，理论上下一次应该 p1 执行。

实际情况是轮到 p1 执行吗？为什么？

不会轮到 p1，而是继续选择 p2。

原因：
p2 执行一个时间片后，其 stride 加上 pass：250 + 10 = 260。由于 stride 用 8 位无符号整型存储，260 溢出后实际值为 260 mod 256 = 4。此时：

  • p1.stride = 255（未变）

  • p2.stride = 4

调度器选择 stride 最小的进程，即 p2（4 < 255），因此 p2 会再次被调度。
这就是 stride 溢出的典型问题：小值因溢出变得极小，破坏了“小 stride 应优先”的预期。

我们之前要求进程优先级 >= 2 其实就是为了解决这个问题。可以证明，在不考虑溢出的情况下, 在进程优先级全部 >= 2 的情况下，如果严格按照算法执行，那么 STRIDE_MAX – STRIDE_MIN <= BigStride / 2。

为什么？尝试简单说明（传达思想即可，不要求严格证明）。

设 pass = BigStride / priority，由于 priority ≥ 2，因此 pass ≤ BigStride / 2。

调度算法每次选 stride 最小的进程 P，将其 stride 增加 pass。假设某一时刻存在 max - min > BigStride / 2，那么选中的 min 加上 pass 后：

text

min + pass ≤ min + BigStride/2 < max

即新的 stride（原 min+pass）仍然小于 max，因此 max 永远无法被选到，与“所有 runnable 进程最终都会被调度”矛盾。
所以差值不可能超过 BigStride / 2。

这个性质确保了在无符号整数溢出发生时，仍能通过比较差值（而非直接数值）来正确判断大小。

  • 作用：root inode（通常 inode 编号为 1）是文件系统的根目录（/）对应的索引节点。它记录了根目录的元数据（类型、权限、大小等）以及指向其数据块的指针，数据块中存储了根目录下的目录项（子文件/子目录的名称与 inode 映射）。整个文件系统的路径解析都从 root inode 开始。

  • 损坏后果：若 root inode 的内容损坏，例如类型不再是目录、数据块指针错误或丢失，则系统无法正确找到 / 下的任何文件或目录。具体表现包括：

    无法执行路径查找（如 open("/file") 会失败）。

    系统启动后可能无法加载初始化程序（如 initproc 无法执行），导致恐慌或死机。

    即使能部分读取，得到的目录项可能错乱或无效，文件系统整体不可用。
    因此 root inode 是文件系统最关键的元数据之一，必须保证其完整性。

第八章功能总结

本次实验实现了多线程环境下的死锁检测机制。通过新增系统调用 enable_deadlock_detect 允许进程按需开启检测功能。维护了三个核心数据结构：Available（可用资源向量）、Allocation（已分配矩阵）、Request（需求矩阵）。在 mutex_lock 和 semaphore_down 操作前，若检测到当前资源分配将导致系统进入不安全状态（即可能死锁），则拒绝操作并返回 -0xDEAD。算法采用经典的银行家算法思想：遍历线程，模拟资源释放，判断是否所有线程都能完成。

第八章问答作业

  1. 在我们的多线程实现中，当主线程 (即 0 号线程) 退出时，视为整个进程退出，
    此时需要结束该进程管理的所有线程并回收其资源。

  • 需要回收的资源有哪些？

  • 其他线程的 struct thread 可能在哪些位置被引用，分别是否需要回收，为什么？

需回收的资源：

  • 所有线程的 内核栈（kstack）

  • 所有线程的 trapframe 页

  • 所有线程的 线程控制块（struct thread）

  • 所有 mutex 和 semaphore 对象（若未释放）

  • 进程页表、文件描述符表、当前工作目录等进程级资源

  • 死锁检测相关的 resource_state 结构及内部数组

其他线程的 struct thread 可能被引用的位置：

  • 调度器的运行队列（runnable 队列）—— 需要回收，需从队列中移除。

  • 锁或信号量的等待队列 —— 应将这些线程从等待队列中踢出，否则悬空指针。

  • curr_proc() 当前运行的某线程（主线程触发退出时，其他线程可能正在睡眠）—— 需要强制唤醒并回收。

  • 系统调用参数中传递的线程 ID（如 waittid）—— 这些只是数值，不直接持有指针，但回收后该 ID 失效。

回收策略：

  • 遍历所有线程，逐个调用 thread_free。

  • 在 thread_free 中，需检查该线程是否在等待队列中，若是则从队列中删除。

  • 从全局线程数组中移除，并释放其内核栈、trapframe 等资源。

  1. 对比以下两种 mutex_unlock 中阻塞锁的实现，二者有什么区别？这些区别可能会导致什么问题？
    （假设无论哪种实现，对应的 mutex_lock 均正确处理了 m->locked）

v1 实现流程：

  1. 先将 m->locked 置 0（表示锁已释放）

  2. 从等待队列中取出一个线程

  3. 若取到线程，则将其状态设为 RUNNABLE 并加入调度队列

v2 实现流程：

  1. 从等待队列中取出一个线程

  2. 若取不到（队列为空），则将 m->locked 置 0

  3. 若取到，则唤醒该线程（不修改 locked）

主要区别：

  • locked 清零的时机不同：v1 在唤醒前清零；v2 仅在队列为空时清零，否则保持 locked=1。

  • 语义上，v1 的锁一旦解锁，即使还有线程在等待，锁状态立即变为“未锁定”，后续竞争者和被唤醒者同时竞争；v2 的锁在还有等待者时保持锁定状态，直接将锁“移交”给被唤醒的线程。

可能导致的问题：

  • v1 问题：
    当锁被释放且存在多个等待线程时，locked=0 后，被唤醒的线程和新来的竞争者可能同时获取锁，造成锁的公平性丧失（有可能新来的线程抢在被唤醒线程之前获得锁，导致被唤醒线程再次阻塞）。极端情况下，饥饿现象可能发生。

  • v2 问题：
    如果被唤醒的线程在调度之前死亡或被取消，锁将永远无法被释放（因为 locked 仍为 1，且队列中已没有其他线程），导致锁死锁。此外，若被唤醒线程在获得 CPU 之前再次被抢占，其他等待者无法获取锁，效率较低。

推荐实现：通常使用 v1 或更常见的“先唤醒后开锁”策略（如先将 m->locked 置 0 再唤醒），并配合调度策略缓解饥饿问题。
